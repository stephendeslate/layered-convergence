# Monitoring Specification

## Overview
Layer 8 monitoring infrastructure providing structured logging,
request correlation, health checks, metrics, and error tracking.

## Structured Logging (Pino)
JSON-formatted log output via Pino logger integrated with NestJS DI.
Log levels: error, warn, info, debug. No console.log in production code.
- VERIFY:AE-MON-03 — Pino logger injectable via NestJS DI
- VERIFY:AE-MON-11 — Pino structured logging in bootstrap

### Log Entry Format
Generated by formatLogEntry from shared package:
{ timestamp: ISO-8601, level: string, message: string, context?: object }
- VERIFY:AE-MON-02 — formatLogEntry produces JSON log strings with timestamp
- VERIFY:AE-MON-07 — createCorrelationId and formatLogEntry exported from shared

## Correlation IDs
Every request receives a unique correlation ID via middleware.
- If client sends X-Correlation-ID header, it is preserved
- Otherwise, a UUID v4 is generated via createCorrelationId from shared
- Correlation ID appears in response header and all log entries
- VERIFY:AE-MON-01 — createCorrelationId generates UUID v4 correlation IDs
- VERIFY:AE-MON-04 — Correlation ID middleware preserves or generates X-Correlation-ID

## Health Endpoints
Both exempt from auth guards and rate limiting (@SkipThrottle).
- VERIFY:AE-MON-09 — Health endpoints exempt from auth and rate limiting

### GET /health
Returns: { status: 'ok', timestamp: ISO-8601, uptime: seconds, version: string }
Version from APP_VERSION environment variable.
Matches Dockerfile HEALTHCHECK path.

### GET /health/ready
Checks database connectivity via Prisma $queryRaw(SELECT 1).
Returns: { status: 'ok'|'error', database: 'connected'|'disconnected', timestamp }

## Request Logging Middleware
Logs every request with: method, URL, status code, duration, correlationId.
Uses structured Pino logger (not console or process.stderr).
- VERIFY:AE-MON-05 — Request logging middleware with all required fields

## Metrics Endpoint
### GET /metrics
Returns in-memory operational metrics:
- requestCount: total requests served
- errorCount: total error responses
- averageResponseTime: mean response time in ms
- uptime: seconds since server start
- VERIFY:AE-MON-08 — In-memory metrics service

## Error Tracking
Global exception filter catches all unhandled exceptions.
Logs: full stack trace, correlation ID, request context.
Returns sanitized error: { statusCode, message, timestamp, correlationId }
No stack traces exposed to clients in any environment.
- VERIFY:AE-MON-06 — Global exception filter logs stack+correlationId
- VERIFY:AE-MON-12 — Monitoring tests for health, correlation, and errors

## Frontend Error Boundaries
Error boundaries log errors with structured format via formatLogEntry.
Includes component name, error message, and digest in context.
- VERIFY:AE-MON-10 — Monitoring module aggregates all monitoring components

## Alerting Thresholds (Recommended)
- Error rate > 5% over 5 minutes
- Response time p95 > 2000ms
- Health endpoint failure for > 30 seconds
- Database connectivity loss (health/ready returns error)
Cross-reference: api.md for endpoint error response format.
